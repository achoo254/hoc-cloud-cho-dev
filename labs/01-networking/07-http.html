HTTP & HTTPS — Request/Response từ dây đến app

Câu hỏi thực tế: "Tại sao curl -v cho thấy request đẹp mà server vẫn trả 400/500? HTTPS làm gì khác HTTP? HTTP/2 nhanh hơn thế nào?"

Không học HTTP chỉ để biết method. Mục tiêu: debug được "400 Bad Request / 401 Unauthorized / 502 Bad Gateway" — phải hiểu cấu trúc request/response, status code phân loại thế nào, và TLS handshake hoạt động ra sao để không bị mù khi có sự cố production.

TL;DR HTTP & HTTPS trong 1 bảng

Cột Why quan trọng nhất — đây là lý do tồn tại của từng khái niệm.

Core HTTP/1.1 vs HTTP/2 vs HTTP/3

Mỗi phiên bản giải quyết vấn đề của phiên bản trước. Hiểu bước tiến để biết dùng khi nào.

Đặc điểm HTTP/1.1 HTTP/2 HTTP/3 (QUIC)
Transport TCP TCP UDP (QUIC)
Format Plain text Binary frames Binary (QUIC streams)
Multiplexing Không (1 req/conn, pipelining không thực dụng) Có — nhiều stream song song trên 1 TCP conn Có — mỗi stream độc lập, không block nhau
Head-of-line blocking Nặng (cả conn chờ 1 req) Vẫn có ở TCP level Không — QUIC giải quyết tận gốc
Header compression Không (plain text, lặp lại) HPACK (delta + Huffman) QPACK (HPACK cải tiến cho QUIC)
TLS bắt buộc? Không (dùng được port 80) Thực tế: có (browser chỉ hỗ trợ H2 qua TLS) Có (QUIC tích hợp TLS 1.3)
0-RTT connect Không Không Có (resume session không cần handshake)
Why quan trọng: Khi thấy site chậm, load nhiều resource (JS/CSS/image), kiểm tra F12 → Network → Protocol. Nếu vẫn là h1 thì đó là bottleneck. HTTP/2 giảm latency đáng kể với web nhiều asset nhờ multiplexing.

Core HTTPS = HTTP + TLS — Handshake hoạt động thế nào?

TLS 1.3 handshake chỉ cần 1 RTT (nhanh hơn TLS 1.2 cần 2 RTT). Hiểu flow để debug "SSL certificate error" và "handshake timeout".

1
ClientHello
Client gửi: phiên bản TLS hỗ trợ, danh sách cipher suites, random nonce, SNI (Server Name Indication — tên domain để server chọn đúng cert khi multi-tenant).
↓
2
ServerHello + Certificate + ServerKeyExchange
Server chọn cipher suite, gửi random nonce của server + certificate (chứa public key + chữ ký của CA). Client verify cert: CA có tin cậy không? Domain khớp SNI không? Cert còn hạn không?
↓
3
Key Exchange (ECDHE)
Dùng Diffie-Hellman trao đổi key: cả 2 bên tạo shared secret mà không bao giờ truyền key qua wire. Đây là lý do HTTPS không bị MITM nghe lén dù traffic đi qua nhiều router.
↓
4
Finished (cả 2 bên)
Tạo session keys từ shared secret + 2 random nonce. Mỗi bên gửi "Finished" message để xác nhận handshake thành công. Từ đây HTTP data được mã hoá bằng AES-GCM (symmetric).
↓
5
HTTP Request/Response đã mã hoá
Từ đây mọi HTTP message đều wrapped trong TLS record. Wireshark thấy "Application Data" — không đọc được nếu không có session key (SSLKEYLOGFILE).
Why học handshake: Khi thấy "SSL_ERROR_HANDSHAKE_FAILURE" cần biết bước nào fail — cert invalid (bước 2), cipher mismatch (bước 1-2), network drop giữa chừng (bước 3). openssl s_client -connect host:443 hiện từng bước để debug.
