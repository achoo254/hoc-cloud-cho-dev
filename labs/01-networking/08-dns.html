DNS — Domain Name System

Câu hỏi cốt lõi: "Gõ example.com, máy tính biết IP nào để kết nối?"

Không học thuộc lòng các loại record. Mục đích: hiểu được tại sao website không load (DNS fail?), tại sao đổi IP phải đợi (TTL), tại sao email bị spam (thiếu SPF/DKIM). DNS là nền tảng của mọi thứ trên internet.

TL;DR DNS trong 1 bảng

Mỗi dòng giải thích tại sao nó tồn tại. Đây là bức tranh toàn cảnh.

Why phân tán: Không có 1 server nào có thể chứa hàng trăm triệu tên miền và xử lý hàng tỷ query/ngày. DNS chia thành cây để mỗi node chỉ quản lý 1 phần nhỏ. Root server chỉ biết TLD, TLD chỉ biết nameserver của domain, nameserver domain mới biết record cụ thể.

TTL và Caching

TTL thấp (60–300s)
5 phút
Dùng khi sắp đổi IP — cache hết nhanh, mọi người thấy IP mới sớm hơn.
TTL cao (86400s)
24 giờ
IP ổn định — resolver cache lâu, giảm query DNS, tăng tốc độ resolve.
Negative Cache
NXDOMAIN
Record không tồn tại cũng được cache. Hỏi lại trong vòng TTL sẽ không tốn thêm query.
Why TTL quan trọng: Đổi IP server mà TTL còn 24h thì người dùng sẽ tiếp tục kết nối IP cũ trong tối đa 24h — dù authoritative server đã cập nhật. Best practice: hạ TTL xuống 300s trước khi đổi IP ít nhất 1 ngày.

UDP 53, TCP 53 và DoH

# UDP 53 — mặc định (query nhỏ ≤ 512 bytes, RFC 1035 §4.2.1) dig example.com @8.8.8.8 # TCP 53 — tự động khi response > 512 bytes hoặc zone transfer (RFC 7766) dig example.com @8.8.8.8 +tcp # DoH — DNS over HTTPS (port 443, encrypted) curl -H 'accept: application/dns-json' \ 'https://cloudflare-dns.com/dns-query?name=example.com&type=A'
Why UDP mặc định: DNS query rất nhỏ, UDP không cần handshake (nhanh hơn). Khi response lớn hơn 512 bytes (giới hạn theo RFC 1035 §4.2.1), server trả TC=1 (Truncated), client kết nối lại bằng TCP theo RFC 7766. DoH encrypt toàn bộ query để ISP không thể sniff tên miền đang truy cập.
